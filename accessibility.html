Accessibility Statement

Effective date: April 1, 2026  ·  Web Everything

Our Commitment

Web Everything is committed to ensuring digital accessibility for people with disabilities. We continually improve the user experience for everyone and apply relevant accessibility standards to help users of all abilities access our website and services effectively.

1. Our Commitment

Web Everything is committed to making our website accessible to people with disabilities. We believe that the web should be open and usable by everyone, regardless of ability, and we design and build with that principle in mind.

Our target conformance standard is the Web Content Accessibility Guidelines (WCAG) 2.1 at Level AA, published by the World Wide Web Consortium (W3C). WCAG 2.1 Level AA is internationally recognized as the benchmark for web accessibility and covers a wide range of recommendations for making web content more accessible, including guidance for users who are blind or have low vision, deaf or hard of hearing, have limited movement, or have cognitive or learning disabilities.

We view accessibility not as a compliance checkbox but as a core quality standard. When we build digital experiences for our clients, we apply the same accessibility thinking we apply to our own properties — because accessible design is good design.

2. Measures Taken

To support accessibility, Web Everything has implemented the following measures on this website:

2.1 Semantic HTML Structure

We use proper HTML5 semantic elements — including <nav>, <main>, <header>, <footer>, <section>, and appropriate heading hierarchy (h1 through h6) — to provide a meaningful document structure that assistive technologies can interpret correctly. Landmark roles and proper heading order help screen reader users navigate the page efficiently.

2.2 Keyboard Navigation

All interactive elements on our website — including navigation links, dropdown menus, form fields, and buttons — are accessible via keyboard alone. Users who cannot use a mouse can navigate the full site using the Tab, Shift+Tab, Enter, Space, and Arrow keys. We maintain a visible focus indicator on all interactive elements so that keyboard users can always see where they are on the page.

2.3 Color Contrast

We design with sufficient color contrast between text and background colors to meet or exceed WCAG 2.1 Level AA requirements (a minimum contrast ratio of 4.5:1 for normal text and 3:1 for large text). We do not rely on color alone to convey meaning — icons, labels, and text are used alongside color to communicate information.

2.4 Alternative Text for Images

All meaningful images on our website include descriptive alternative text (alt attributes) so that screen reader users receive equivalent information. Decorative images that do not convey meaning are marked with empty alt="" attributes so they are correctly ignored by assistive technologies.

2.5 ARIA Labels and Roles

Where native HTML semantics are insufficient, we use Accessible Rich Internet Applications (ARIA) attributes — including aria-label, aria-expanded, aria-controls, role, and related attributes — to provide additional context for screen readers and other assistive technologies. We follow the principle of using ARIA only when native HTML cannot achieve the same result.

2.6 Responsive and Scalable Design

Our website is fully responsive and supports text resizing up to 200% without loss of content or functionality. We use relative units (rem, em, %) for font sizes and layout to ensure that user-set browser font preferences are respected. The site is designed to be usable on a range of screen sizes and viewport configurations.

2.7 Form Accessibility

All form fields on our website have associated visible labels and descriptive placeholder text. Required fields are clearly marked. Error messages are descriptive and associated with the relevant form field so they are announced by screen readers. Our forms do not rely on color alone to indicate errors.

2.8 Animation and Motion

Our website uses CSS animations for visual enhancement. We respect the prefers-reduced-motion media query — users who have indicated a preference for reduced motion in their operating system settings will receive a simplified, non-animated experience that does not trigger vestibular disorders or other motion sensitivities.

3. Known Limitations

Despite our best efforts, there may be some areas of our website that do not yet fully conform to WCAG 2.1 Level AA. We are actively working to identify and address these limitations. Currently known limitations include:

  • Animated SVG graphics — The animated network and icon SVGs used on the homepage and related pages currently lack detailed text descriptions beyond the surrounding content context. We are evaluating approaches to provide more descriptive alternatives for these elements.
  • Mobile navigation — The mobile hamburger menu is currently functional via touch and keyboard, but ARIA expanded/collapsed state announcements are being refined to improve screen reader experience on mobile devices.
  • Third-party embedded content — Any third-party tools or embeds (e.g., scheduling tools, analytics scripts) may have their own accessibility limitations that are outside our direct control. We evaluate third-party tools with accessibility in mind and will replace or remove tools that present material barriers.

We are committed to addressing these limitations in our ongoing development cycle. If you encounter a specific barrier not listed here, please contact us using the information in Section 5 so we can investigate and prioritize a fix.

4. Technical Specifications

The Web Everything website relies on the following technologies for conformance with WCAG 2.1:

  • HTML5 — Semantic markup and document structure.
  • CSS3 — Visual styling, responsive layout, and animation with prefers-reduced-motion support.
  • JavaScript — Progressive enhancement for interactive components; core content is accessible without JavaScript enabled.
  • SVG — Scalable vector graphics for icons and illustrations.

Supported Browsers and Assistive Technologies

Our website is designed and tested to work with the following browsers and assistive technology combinations:

  • Google Chrome (latest) with NVDA on Windows
  • Mozilla Firefox (latest) with NVDA on Windows
  • Apple Safari (latest) with VoiceOver on macOS
  • Apple Safari on iOS (latest) with VoiceOver
  • Google Chrome on Android (latest) with TalkBack
  • Microsoft Edge (latest) on Windows

We aim for functional compatibility across all modern browsers and major screen readers. If you experience difficulty with a specific browser or assistive technology combination, please let us know.

5. Feedback and Contact

We welcome your feedback on the accessibility of the Web Everything website. If you encounter any barriers, discover content that is not accessible, or have suggestions for improvement, please contact us:

  • Email: hello@webeverything.io — please include "Accessibility" in the subject line.
  • Company: Web Everything, operated by Kenny Gimpert II
  • Location: Detroit, MI

We are committed to responding to accessibility feedback within three (3) business days. If the issue requires investigation or remediation, we will provide you with an estimated timeline for resolution and keep you informed of our progress.

We try to respond to feedback promptly. If you need information on our website in a different format — such as an accessible PDF, large print document, or plain text version — please let us know and we will work to accommodate your needs as quickly as possible.

6. Formal Complaints

If you are not satisfied with our response to your accessibility feedback, you have the right to escalate your concern. Depending on your jurisdiction, you may be able to contact:

  • United States: The U.S. Department of Justice (DOJ) Civil Rights Division, which enforces the Americans with Disabilities Act (ADA), or the U.S. Access Board for web accessibility guidance.
  • European Union: Your national market surveillance authority responsible for enforcing the European Accessibility Act (EAA).
  • United Kingdom: The Equality and Human Rights Commission (EHRC), which monitors compliance with the Public Sector Bodies Accessibility Regulations and the Equality Act 2010.

We sincerely hope that any accessibility concern you bring to us can be resolved directly and promptly. We take all feedback seriously and are committed to continuous improvement.

7. Assessment Approach

Web Everything assesses the accessibility of this website using the following approach:

7.1 Self-Assessment

Our primary assessment method is ongoing self-evaluation conducted by our development team. This includes:

  • Manual keyboard navigation testing on new and updated pages.
  • Screen reader testing using VoiceOver (macOS/iOS) and NVDA (Windows) during development.
  • Color contrast checking using browser developer tools and dedicated contrast analysis tools.
  • HTML validity checking against W3C standards.
  • Review of new features and content against WCAG 2.1 Level AA success criteria before publication.

7.2 Automated Testing

We use automated accessibility scanning tools — including axe DevTools and Lighthouse — as part of our development workflow to identify common accessibility issues early. We recognize that automated tools detect only a subset of potential accessibility barriers and use them as a complement to, not a replacement for, manual testing.

7.3 Ongoing Review

We conduct a comprehensive accessibility review of our website on a rolling basis, with a full audit performed at minimum annually or whenever significant changes are made to the site's structure, design, or technology stack. This statement will be updated following each review to reflect the current state of accessibility on our site.
